## 18. Local Inference Acceleration Layer

### Doctrine

Claude remains the primary executive, orchestration, and reasoning brain for AKIOR. Ollama on Apple Silicon, accelerated by the MLX framework, operates as the local inference layer — a speed/cost/latency optimization surface underneath Claude's orchestration.

**MLX does not replace Claude reasoning.** MLX improves local model execution performance on the Mac Mini's Apple Silicon. Ollama provides the runtime that serves local models. Together they form a cost-saving auxiliary engine for tasks where cheap, fast, local execution is the better surface.

### Role Assignment

| Layer | Role | When Used |
|-------|------|-----------|
| **Claude (primary)** | High-complexity reasoning, multi-step planning, tool orchestration, high-stakes outputs, nuanced drafting, architectural decisions | Default for all tasks unless local routing is clearly better |
| **Ollama + MLX (local)** | Classification, summarization, lightweight drafting, local preprocessing, repetitive low-risk tasks, cost-saving support, high-frequency agent subroutines, fallback coding helpers | When task is well-defined, low-risk, and does not require frontier reasoning |
| **Remote GPU / SSH** | Large-model inference, training, heavy computation exceeding Mac Mini capacity | Only for workloads that exceed 24GB unified memory |

### Routing Rule

**Use the cheapest reliable execution surface that still satisfies the task.**

| Priority | Surface | Condition |
|----------|---------|-----------|
| 1 | Ollama local (MLX-accelerated) | Task is lightweight, well-defined, low-risk, and local model quality is sufficient |
| 2 | Claude | Task requires strong reasoning, planning, orchestration, high-trust output, or local model quality is insufficient |
| 3 | Remote GPU / SSH (Threadripper) | Task exceeds Mac Mini hardware capability |

**Automatic escalation:** If local Ollama output quality is insufficient for the task, AKIOR escalates to Claude immediately. AKIOR never gets stuck because a local model is weak.

### Hardware Constraints

The AKIOR experiment runs on Mac Mini M4 Pro with 24GB unified memory and 512GB storage. This hardware class supports efficient small-to-medium local models well. It does not support very large local models as default targets.

**Rules:**
- Default Ollama models: small, efficient models appropriate for 24GB unified memory (e.g., 7B-13B parameter range quantized)
- Do not assume large models (30B+) are stable defaults on this hardware
- Large-model local experiments are optional and must be benchmarked before routine use
- If a local model causes memory pressure, swap thrashing, or latency spikes, downgrade to a smaller model or route to Claude

### Local Inference Task Types

| Task Type | Example | Expected Model Size |
|-----------|---------|-------------------|

| Summarization | Document digest, meeting notes summary | 7B-13B |

| Code completion assist | Simple autocomplete, boilerplate generation | 7B-13B (code-tuned) |
| Data extraction | Structured parsing from unstructured text | 7B |
| Preprocessing | Clean/format data before Claude processes it | 3B-7B |
| Repetitive subroutines | Batch classification, bulk tagging | 3B-7B |

### Capability Status

Ollama is an approved local runtime dependency in the AKIOR stack. MLX-backed acceleration is part of the local runtime performance layer. This addition does not introduce any owner-review bottleneck and does not demote Claude's role. AKIOR installs, configures, benchmarks, and uses Ollama + MLX as part of its standard v1 bootstrap.

## 24. Adapter Hierarchy Doctrine

**Hard rule:** AKIOR must prefer the most deterministic execution surface first.

| Priority | Surface | When to Use | Reliability |
|----------|---------|-------------|-------------|
| **1** | API / connector / MCP | Always first if available for the target app | Highest — deterministic, structured, retryable |
| **2** | CLI / SDK / direct system command | When API not available but CLI/SDK exists | High — deterministic, scriptable, auditable |
| **3** | Deterministic browser automation (Playwright) | When API/CLI not available but page structure is stable | Medium-high — scriptable, but fragile to UI changes |
| **4** | Visual Computer Use (mouse/keyboard/screen) | Last resort when no structured path exists | Medium — functional but variable, requires screenshot evidence |

This is not a reduction in autonomy. It is a reliability multiplier. AKIOR always has access to all surfaces. This doctrine determines which one to try first for maximum success probability.

**Escalation behavior:** If the preferred surface fails, AKIOR escalates to the next surface automatically, logs the escalation, and flags the failure for the app's failure-mode library.

**Inference routing:** For tasks involving model inference (classification, summarization, drafting, code generation), the cost-routing rule from Section 18 applies in parallel: use Ollama local for lightweight tasks, Claude for high-complexity reasoning. These two hierarchies operate together — surface selection (this section) determines how AKIOR interacts with an app; inference routing (Section 18) determines which model brain handles the reasoning.

## 27. State, Session, and Resume Layer

### Checkpointing

AKIOR creates checkpoints at defined stages of multi-step tasks. A checkpoint records: task ID, current step, completed steps, intermediate artifacts, and resume instructions.

Checkpoint triggers: before any irreversible action, after each major phase of a multi-step workflow, and at fixed intervals for long-running tasks.

### Session Restoration

If a session ends unexpectedly (crash, timeout, machine restart), AKIOR reads the latest checkpoint and resumes from the last completed step. No re-execution of completed steps.

### Retry Rules

| Failure Type | Retry Behavior |
|-------------|---------------|
| Transient (network timeout, rate limit) | Wait 30s, retry up to 3 times |
| Auth failure | Re-authenticate, retry once |
| Tool error (CLI/API) | Log error, try fallback surface per Adapter Hierarchy |
| Logic error (wrong output) | Log failure, attempt different approach, max 2 retries |
| Persistent failure | Log to exception queue, flag in canary report |

### Idempotent Actions

AKIOR must design actions to be safely re-runnable. Before creating a record, check if it already exists. Before sending a message, check if already sent. Before deploying, check current state matches expected pre-state.

### Crash Recovery

On Mac Mini restart or unexpected session termination: AKIOR reads `state/checkpoints/` on next startup, identifies unfinished tasks, and resumes or reports them in the morning briefing.

## 29. Canary Jobs and Drift Detection

### Daily Canary Suite

Run every morning before the owner briefing:
- Gmail connector health check (send test → verify receipt)
- GitHub CLI auth check (`gh auth status`)
- Wix login and page-load check
- Scheduled task execution check (did yesterday's scheduled tasks all fire?)
- File system health check (disk space, key directories intact)
- All MCP servers responding
- **Ollama local inference health check** (ping Ollama server, run a 10-token classification task, verify response latency < 2s and memory usage within bounds)

### Weekly Regression Suite

Run every Sunday:
- Full Golden Task Suite (GT-1 through GT-8)
- App Pack validation for each approved app (login, basic flow, evidence capture)
- Canary job pass rate for the past week
- Token cost trend check
- **Local inference benchmark** (run standard summarization + classification + code-completion tasks against Ollama; compare latency and quality to baseline; flag if degraded)

### UI Drift Detection

For browser-automated apps (Wix, Instagram, Canva, QuickBooks): AKIOR takes baseline screenshots of critical pages at App Pack creation time. Weekly, re-capture and compare. If layout changes significantly, flag the App Pack for selector/flow updates before the next production run.

### Credential / Session Health

Daily check that all stored credentials, tokens, and sessions are valid. If any are expired or approaching expiration, refresh proactively. Log all credential events.

## 33. Bounded-95 Doctrine

**This is the most important reliability concept in the SSOT.**

AKIOR targets 95%+ success rate inside the approved app catalog (Section 22) and approved task-family matrix (Section 24).

Outside that boundary, AKIOR may still explore and operate with full autonomy — but those tasks are classified as **exploratory**, not as high-confidence production-like operations.

**What this means in practice:**
- Tasks within the approved boundary are expected to succeed on the first attempt, with automatic retry handling edge cases
- Tasks outside the approved boundary are valuable learning opportunities that may expand the boundary over time
- The approved boundary expands as AKIOR masters new apps (via Golden Task graduation) and new task families (via Task Family documentation)
- Success rate is measured and tracked in the weekly regression report

**Success SLOs by Task Family:**

| Task Family | Target SLO | Measurement |
|-------------|-----------|-------------|
| Customer reply | 95% accuracy (tone, content, recipient) | Correct replies / total reply attempts |
| Landing page build | 90% first-attempt success | Successful deploys / total build attempts |
| PR creation | 95% first-attempt success | Green-CI PRs / total PR attempts |
| Scheduled task execution | 98% fire rate | Tasks fired on schedule / tasks scheduled |
| Social post workflow | 90% first-attempt success | Published posts / total post attempts |

| Retry-to-success | 95% resolution within 3 retries | Resolved issues / total retried issues |

**SLOs are not owner-imposed limits.** They are AKIOR's own operational targets that make the experiment measurable and the system accountable to itself.
